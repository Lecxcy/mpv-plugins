# enhanced-ab-loop SPEC

本文档描述 `enhanced-ab-loop`(并入了 `tail-frame-extension` 的尾帧冻结能力)
这个 C++ mpv 插件的行为约定。§0-§7 是**功能规格**，描述插件现在实际的行为，
供审计代码、编写测试、或者从零重新实现时对照；§8 是**这一轮实现踩过的坑**——
每一条都带现象、复现步骤、根因、修复方式，目的是让"从零重新实现一遍"这件事
不再重复踩同样的坑；§9 是已知限制（不是 bug）。

用户文档见 [README.md](README.md)；手动测试清单见 [TESTING.md](TESTING.md)。

对应开发分支：`plugin/enhanced-ab-loop-rewrite`(从 `main` 切出，遵循
AGENTS.md「编写某个插件时应先创建独立分支」的约定)。

## 0. 背景与目标

重写前的 `plugins/enhanced-ab-loop/lua-reference/enhanced-ab-loop.lua` 是纯
Lua 实现，有几个已确认的问题：

- 用 `mp.add_periodic_timer(0.05, check)` 轮询 `time-pos` 判断是否越过循环
  终点，精度受 50ms 定时器粒度和脚本事件循环调度双重制约。
- 完全没有使用 mpv 原生的 `ab-loop-a`/`ab-loop-b`/`ab-loop-count` 属性。
- 循环终点无法设到 `tail-frame-extension` 用 `tpad`/`apad` 滤镜补出来的
  尾部延长段里，因为 `duration` 属性和 demuxer 可 seek 范围不受滤镜链影响。
- `sort_segments` 的比较函数用 epsilon 容差做 tie-break，不满足严格弱序，
  `table.sort` 在这种情况下不保证正确排序。
- `eof-reached` 处理器硬编码跳到 `active[1].a`(整体列表里最靠前的一段)。

重写目标（已全部完成）：

1. 用 mpv 原生 `ab-loop-a`/`ab-loop-b` 的帧级 pts 比较驱动跳转，替代 Lua
   定时器轮询，同时支持多段不相交区间的顺序循环。
2. 把"循环到文件真实末尾时冻结尾帧"的能力并入本插件，替代
   `tail-frame-extension` 原来单独存在但无法与 AB-loop 配合的状态。
3. 修掉排序、eof 跳转目标这两个已确认的 bug。
4. 支持挑选已有区间的子集参与循环(启用/禁用，按位置顺序)。
5. 支持把区间列表存档/读档。
6. 明确"插入新区间"和"调整已有区间边界"这两类操作的边界，避免语义模糊。

实现语言 C++：纯逻辑部分(排序、重叠检测、active 队列计算、序列化)拆到
`include/enhanced_ab_loop/` + `src/logic.cpp`/`src/store.cpp` 里的纯函数，
配 Catch2 单元测试（`tests/`，不依赖 mpv，可以直接跑）；mpv 交互部分(属性
读写、事件监听)在 `src/plugin.cpp` 里。按键绑定参照 `enhanced-drag` 的写法
接住 `key-binding` client message（`mp.add_key_binding` 在 C 插件这层没有
直接等价物）。

## 1. 数据模型

### 1.1 完整区间(segment)

```
segment = {
    a: double,       -- 起点，单位秒，直接存 mpv 报告的原始 double，不做量化
    b: double,       -- 终点，同上
    enabled: bool,   -- 是否参与循环(默认 true)，见 §5
}
```

`segments` 是这些完整区间的列表，任何时刻都保持互不重叠、按 `a` 升序排列
(`sort_segments`)。

**精度处理**：

- 不引入帧号或定点整数量化。double 本身精度远超任何实际帧时长量级。
- **存储值之间的比较**(排序、检测两个已定义 segment 是否重叠)一律用精确
  相等/大小比较，不用 epsilon(`overlaps_complete`)。
- **实时播放位置 vs 已存储边界的比较**(判断光标当前是否"落在"某个已有
  完整区间内，用来触发 §3 的 Move A/B、extend、toggle-segment 判定)保留
  一个固定的、故意放宽的 epsilon 常量 `kCursorEpsilon = 0.005`，这是给键盘
  操作/原生 seek 落点留出的人体工学余量，不是精度补偿。

### 1.2 待定端点(pending point)

同一时刻最多存在一个：

```
pending = { has_a: true, a: double } | { has_b: true, b: double } | 都为 false
```

对应"只设了一个端点、还没配对成完整区间"的状态。不是一个独立的模式标志
位，只是 `Pending` 结构体里恰好一个字段非空。

### 1.3 循环开关

`loop_enabled: bool`，由 `|` 键(`toggle-enabled`)控制：

- `set-a`/`set-b`/`unset-a`/`unset-b`/nudge/extend/`toggle-segment` 这些
  **编辑动作本身**任何时候都可以执行，不受 `loop_enabled` 影响，`segments`/
  `pending` 的内容变化完全独立于循环开关。
- 但"编辑动作需要先把光标 seek 到目标位置"这个前置步骤，在 `loop_enabled
  = true` 时会受 §2.6 的"孤立光标吸入"规则影响——见 §8 第 7 条，这是一条
  **有意保留的使用限制**，不是 bug：循环开着时只能舒服地调整"光标已经站
  在里面"的那一段，没法顺手 seek 到空白处插入全新的一段。
- `loop_enabled = false` 时不发生任何自动跳转(包括对 `pending` 中"开放
  区间"的实时循环)，但已经设置的端点、`segments` 内容保持不变。
- `loop_enabled = true` 时，`segments` 中 `enabled = true` 的区间参与循环，
  `pending`(如果存在)也会作为一个临时的"开放区间"实时参与，见 §2.5。

## 2. 核心跳转机制

### 2.1 原生驱动，取代定时器轮询

不用 `add_periodic_timer` 轮询判断，改为动态写 mpv 原生
`ab-loop-a`/`ab-loop-b` 属性，让 mpv 内核用帧级 pts 比较触发跳转：

- `playloop.c` `handle_loop_file` 在到达 EOF 判定时，读取**此刻**的
  `ab-loop-a`(经过 §2.2 排序后)实时值作为 seek 目标，`MPSEEK_EXACT` 精确
  seek。
- `video.c`/`audio.c` 对每一帧比较 `pts >= 有效 ab-loop-b`，一旦命中该帧
  直接判 EOF(不渲染)。
- 两处都是"触发那一刻才读当前属性值"，不是缓存值。
- `ab-loop-count` 保持默认 `inf`，不需要按段管理计数
  (`misc.c`:`remaining_ab_loops` 对 inf 值不递减)。

### 2.2 硬约束：ab-loop-a/ab-loop-b 是无序的 [min, max] 区间对，不是两个独立语义的属性

**这是整个跳转机制里最容易设计错的一点，务必先读这一节再动手实现 §2.3。**

mpv 的 `get_ab_loop_times()`(`misc.c`)在使用 `ab-loop-a`/`ab-loop-b` 之前，
恒定会做一次 `MPSWAP`，保证返回的 `ab[0] <= ab[1]`：

```c
t[0] = opts->ab_loop[0];
t[1] = opts->ab_loop[1];
...
if (t[0] * dir > t[1] * dir)
    MPSWAP(double, t[0], t[1]);
```

之后不管是 `get_play_end_pts`(算播放截止点，用 `ab[1]`——较大值)还是
`handle_loop_file`(算命中截止点后的 seek 目标，用 `ab[0]`——较小值)，用的
都是**排序之后**的值。也就是说：

- **截止点永远是数值较大的那个**，**跳转目标永远是数值较小的那个**，跟
  你把这两个值字面上写进 `ab-loop-a` 还是 `ab-loop-b` 完全无关。
- 这条规则不接受任何例外——不存在"只要顺序写对就能让 mpv 按字面属性名
  解释"这种做法。

**直接后果**：不能设计成"ab-loop-a = 我想让原生机制跳到的下一个目标(可能
比当前段终点大)、ab-loop-b = 当前段自己的终点"这种"预置下一跳"的写法。
只要目标值比终点值大——多段**正向**轮播里下一段的起点几乎总比当前段的
终点大——排序就会把这两个值反过来解释：截止点变成了"下一段起点"(播过头
很久才触发)，跳转目标变成了"当前段自己的终点"(倒着跳)。§8 第 3 条记录了
这条坑第一次被踩中时的具体现象。

**唯一总能正确工作的写法**：`ab-loop-a`/`ab-loop-b` 只表达"当前落在的这一
段自己的 `a`/`b`"，天然 `a < b`，不可能被排序坑。见 §2.3。

### 2.3 单段自环 + 显式跨段补跳

`compute_jump_pair(order, landed_index, real_duration, tail_freeze_seconds)`
(`logic.h`/`logic.cpp`)只返回**当前落在的这一段自己的** `a`/`b`：

- `landed_index` 对应项的 `b` 有值：`ab_loop_a = current.a`，`ab_loop_b =
  current.b`(尾帧冻结调整见 §4)。
- `landed_index` 对应项的 `b` 是 `nullopt`(开放到文件末尾)：两者都不写，
  交给 §2.4 的 eof-reached 兜底。

原生机制只负责"在这一段自己的 `a`/`b` 之间自环"——命中 `b` 就跳回自己的
`a`，不会自己前进到下一段。真正的"跳到下一段"由 `plugin.cpp` 的状态机
显式补一次 `seek` 完成：

- `PluginState::armed_index`：当前 `ab-loop-a`/`ab-loop-b` 对应
  `active_order` 里哪一项自己的 `a`/`b`。`refresh_loop`(所有编辑动作—— 
  `set-a`/`set-b`/nudge/extend/unset/`toggle-segment`/`toggle-enabled`/
  文件加载——都会调用它)每次都无条件按"光标当前落在哪一项"刷新这个值，
  不带任何跨段跳转判断——这只是"预置好当前该看住哪一项"，判断"是不是该
  跳到下一项了"是下面 `on_native_landing` 专门的职责，两者不能混在一起，
  否则编辑操作里凑巧把光标停在某项自己的 `a` 上(比如循环刚跳回来落在边
  界附近再手动调整)会被误判成"这一项播完了，跳下一项"。
- `on_native_landing`(只从 `MPV_EVENT_SEEK`/`MPV_EVENT_PLAYBACK_RESTART`
  调用)：判断这次落地是不是"原生自环刚好命中了 `armed_index` 自己的
  `a`"——是的话说明这一项已经完整播完一轮，算出 `active_order` 里的下一项
  (`(idx+1) % order.size()`，wrap 回第一项)，显式发一次
  `seek <下一项.a> absolute exact`。判定"播完了"需要同时满足：
  1. 不是我们自己刚发出去的补跳/reengage seek 的回执(见 §2.4 的事件去重)。
  2. 落地的项下标等于 `armed_index`，且落地位置精确等于(±`kCursorEpsilon`)
     这一项自己的 `a`。
  3. 这一项有真正的 `b`(开放到文件末尾的不适用)。
  `next == armed_index`(`active_order` 只有一项，自己套自己)时不需要真的
  发 `seek`——已经就在正确的位置上，直接落到 `refresh_loop` 重新武装同一
  项即可，省掉一次没必要的 seek 往返(单段循环最在意"跳转瞬间不能有卡顿"，
  多一次 `seek` 就是多一次风险)。

### 2.4 事件去重：SEEK 和 PLAYBACK_RESTART 数量不对称

mpv 对"插件自己主动发出的显式 `seek` 命令"(跨段补跳、§2.6 的 reengage、
`eof-reached` 兜底)**恒定**会发两条事件通知——`MPV_EVENT_SEEK`(排队)和
`MPV_EVENT_PLAYBACK_RESTART`(真正落地完成)；而**原生 ab-loop 自己内部**
触发的自环跳转**只发一条** `MPV_EVENT_SEEK`，没有 `PLAYBACK_RESTART`(它
不走完整的"重新开始播放"流程，只是 demux 悄悄挪了一下)。

`PluginState::pending_self_seek`：true 表示上一次的 `seek` 是插件自己为了
跨段补跳/reengage 发出去的，还没等到它落地确认。**必须同时等到 `SEEK` 和
`PLAYBACK_RESTART` 两条都收到才能清掉这个标记**，中途收到的 `SEEK` 先按
兵不动——只有 `PLAYBACK_RESTART` 才代表这次自己发的 seek 真正落地，这时
才清掉标记、调 `refresh_loop` 确认武装。§8 第 5 条记录了"只等第一条就清
掉标记"会导致什么后果(暂停状态下的 seek 死循环)。

### 2.5 eof-reached 兜底

`get_ab_loop_times`(`misc.c`)要求 A、B 都必须是有效数值，`pending` 只有
一个端点时无法直接表达给原生机制，所以 `pending` 只有一个端点、且不存在
下一个 active 区间兜底时(即 `pending` 是 `active_order` 中最后一项且 `b`
未定)，会真正播到文件末尾触发 `eof-reached`，而不是被原生 pts 截断拦下。

`eof_fallback_target(order)` 返回 `active_order` 里排在**最后**的那一项
自己的 `a`(不是硬编码第一项)——真正触发 `eof-reached` 的必然是这一项，
因为更早的项都有自己的 `b` 会提前用原生 pts 截断拦下。`on_eof_reached`
发出这次 seek 时也要标记 `pending_self_seek = true`(同 §2.4)。

### 2.6 pending 的临时循环边界

`pending` 存在时(§1.2)，需要一个临时边界让它在编辑过程中也能实时循环
(不能等到真正设置第二个端点才开始循环，用户需要靠反复播放来找到另一个
端点该落在哪)，见 `build_active_order`：

- 只有 `a`、没有 `b`：临时终点 = **下一个**完整区间的**起点**(不是终点)，
  如果不存在下一个完整区间，临时终点为空(即循环到文件真实末尾，由 §2.5
  的 eof-reached 兜底处理)。
- 只有 `b`、没有 `a`：临时起点 = **上一个**完整区间的**终点**(不是起点)，
  如果不存在上一个完整区间，临时起点为 `0`。

这里特意用"近端"(下一个区间的起点/上一个区间的终点)而不是"远端"，是
为了避免编辑中的开放区间一路播穿整个相邻的已有区间——只应该在相邻区间
开始之前打转，不应该吞掉相邻区间本身的循环范围。查邻居时用的是完整
`segments` 列表(含被禁用的)，不是只看 `enabled` 的 order——被禁用的区间
依然真实占用那段时间，不应该被当成不存在。

这个临时边界**只影响 pending 状态下的实时循环范围**，不影响它最终配对
成完整区间时的实际取值(那部分见 §3.1)。

### 2.7 孤立光标的吸入：`loop_reengage_target` / `try_reengage_seek`

`locate_active_index(order, pos)` 定位"当前光标应该算落在 `active_order`
的哪一项"：

- `pos` 落在某一项自己的 `[a, b)` 范围内(`is_within_active_entry`，`b` 为
  空时只要求 `pos >= a`)：返回那一项。
- 不在任何一项内部，但存在"起点比 `pos` 大"的项(空隙里，含全部区间最
  前面)：返回**下一个**要开始的那一项。
- 全部都在 `pos` 之前(含恰好落在最后一项自己的 `b` 上)：返回**最后一项**
  自己(不是 wrap 回第一项——这条兜底方向本身是 §8 第 2 条踩过的坑，务必
  保持"返回最后一项"，不要改回"返回第一项")。
- `order` 为空：`nullopt`。

`loop_reengage_target(order, pos)` 基于上面的定位，判断"光标是不是已经
落在某个激活区间内部"：

- 已经在里面：`nullopt`(不需要额外动作，继续播放就会正常触发 §2.3 的
  自环)。
- 不在里面(空隙里，或者已经越过了全部区间)：算出"应该立刻 seek 到哪"——
  `pos` 还没到定位出来的那一项自己的 `a`：目标就是那一项自己的 `a`；
  `pos` 已经越过了那一项自己的 `a`(意味着已经越过全部区间)：目标改成
  `active_order` 里下一项(wrap 回第一项)自己的 `a`。

**为什么不能指望"继续播放，碰巧落回循环范围"**：mpv 的 `update_ab_loop_
clip()`(`playloop.c`)只要发现当前 `pos` 已经超过(排序后的)有效终点，就
会直接把 `ab_loop_clip` 判 `false`，原生循环机制整个不介入——继续播放只
会一路播到文件真正结束(配合 `loop-file=inf` 就表现成"从头重播"，不是回
到定义的循环里)。`try_reengage_seek`(`plugin.cpp`)包一层：`loop_enabled`
为 `false` 时直接不做任何事；否则调用 `loop_reengage_target`，有目标就
显式发一次 `seek`(并标记 `pending_self_seek`，同 §2.4)，返回是否发出了
这次 seek。

**五个会让光标"孤立"(不再被任何激活区间覆盖)的入口，都必须调用
`try_reengage_seek`**——这是一条贯穿多个操作的通用规则，不是某个具体动
作的特例(§8 第 6 条记录了这条规则是分四轮才补全的教训)：

1. `on_toggle_enabled`：打开循环开关的那一刻，光标可能停在关闭期间留下
   的任意位置。
2. `on_native_landing` 的兜底分支(判定"不是原生自环播完"之后)：覆盖手动
   seek(拖进度条等不是走 `set-a`/`set-b` 的操作)落在空隙里的情况。
3. `on_unset_a` / `on_unset_b`：撤销端点可能让光标所在的这一段彻底消失。
4. `on_toggle_segment`：禁用光标所在的这一段会把它从 `active_order` 里
   摘掉。
5. `apply_loaded_segments`(`on_load_loops` 精确命中，或 `on_confirm`
   确认疑似改名之后)：存档里加载出来的 `segments` 跟光标当前位置大概率
   对不上——加载前光标停在哪完全是加载前的历史状态决定的，跟存档里的
   区间没有任何关系。

`set-a`/`set-b` 本身**不**调用 `try_reengage_seek`——它们的语义就是"把
当前光标位置记录成一个新端点"，落地位置理所当然可能不在任何已有区间
内部(否则不可能插入新区间)，这里如果也做"落在空隙就吸入"会直接破坏插
入新区间的工作流(见 §8 第 7 条)。`nudge`/`extend` 同理不调用——它们的
触发前提本身就要求光标已经在正确的语义位置上(区间内部/空隙里)，操作
结果不会让光标脱离这个位置太远。

## 3. 插入 / 调整区间的交互

### 3.1 插入新区间(set-a / set-b)

`set-a`/`set-b` 在光标不落在任何已有完整区间内部时触发，语义**始终且仅
表示"插入一个独立的新区间"**，不做任何隐式的边界借用/吞并：

- 第一次按(比如 `set-a`)记录 `pending.a`。
- 第二次按另一个键(`set-b`)尝试配对成完整区间，校验：
  - `a < b`(否则拒绝，提示非法)。
  - 与 `segments` 中任意已有完整区间不重叠(`overlaps_complete`，精确比较，
    不用 epsilon)。命中重叠直接拒绝、提示冲突，不做任何自动收缩/clamp/
    忽略邻居的行为。
- 校验通过才真正插入 `segments`，按 `a` 排序，清空 `pending`。

如果最终两个端点落在已有区间内部导致重叠，用户需要自己把端点往回调再
重试，插件不代为决策。

### 3.2 调整已有区间边界

两类相邻但不同的操作，用不同的触发方式区分，不靠事后结果反推意图：

- **Move A / Move B**(对齐 PotPlayer)：光标落在某个已有完整区间**内部**
  时按 `set-a`/`set-b`，直接把该区间自己的 `a`/`b` 移动到光标位置(前提
  是移动后不会和相邻区间冲突、不会导致区间反转——由于互不重叠的不变式，
  这个前提天然满足，不需要额外校验)。
- **Extend segment**(`extend-prev`/`extend-next` 两个独立键位，不复用
  set-a/set-b)：光标落在空隙(不在任何完整区间内部)时触发，拉伸相邻区间
  的近端边界到光标位置：
  - `extend-prev`：把光标所在空隙**前面**那个已有区间的 `b` 拉伸到光标
    位置；空隙前面没有区间(光标在第一段之前)则拒绝，提示"没有前一段
    可拉伸"。
  - `extend-next`：把光标所在空隙**后面**那个已有区间的 `a` 拉伸到光标
    位置；空隙后面没有区间则拒绝，提示"没有下一段可拉伸"。
  - `pending`(§1.2)非空时(正在插入一段还没配对完成)，extend 直接拒绝，
    提示需要先完成或放弃当前插入——同一时间只允许一种进行中的编辑。
  - 拉伸只会让边界往空隙方向移动，不会导致 `segments` 顺序颠倒，成功后
    不需要重排，但仍需触发 §2.3 的"重新计算 active 队列 + 重新武装"。

  set-a/set-b 永远只处理插入；Move A/B 的前提是光标在区间内部；extend 的
  前提是光标在空隙里；三者的触发条件两两不重叠，同一个光标位置只可能
  命中其中一种。

### 3.3 nudge

`nudge-a-*`/`nudge-b-*`(按固定步长 `kNudgeStep = 0.05` 微调光标所在完整
区间的端点)：

- **触发条件比 Move A/B 宽松**：只要求光标落在 `[a, b]` 闭区间内即可
  (`is_within_closed_range`)，不要求距边界超过 epsilon——循环刚跳回
  `a`/`b` 附近(原生 `MPSEEK_EXACT` 落点通常就在边界几毫秒内)时，恰恰是
  最想 nudge 的时机，不能被拒绝。
- **顶到边界时拒绝，不做静默 clamp**：跟 §3.1/§3.2 一致的"顶到边界就拒绝
  这次 nudge、提示已到边界"。

调整成功后需要触发 §2.3 的"重新计算并武装"。

### 3.4 撤销端点(unset-a / unset-b)

把某个完整区间拆回一个 `pending`：

- `unset-a`：光标落在某个完整区间内部，把这个区间从 `segments` 里移除，
  它的 `b` 变成 `pending.b`(`pending.has_a = false`)。
- `unset-b`：对称，`a` 变成 `pending.a`。
- 拆出来的 `pending` 不嫁接到相邻完整区间上、不篡改相邻区间自己的显示；
  展示位置见 §3.5。
- 待定状态下(`pending` 已经只剩一个端点)再按对应的键：`unset-a` 在
  `pending.has_a` 为 false 时拒绝(`"A not set"`)；`unset-b` 同理。如果
  `pending` 里当前那个字段确实有值(比如 `unset-a` 之后 `pending.has_b`
  为 true，再按 `unset-b`)，会把 `pending` 也清空——这是"彻底删除一个
  区间"的完整操作序列：先 `unset-a` 把区间拆成 `pending{b}`，再
  `unset-b` 把这个 `pending.b` 也清掉，最终 `pending` 归零、区间彻底
  消失。
- 撤销可能让光标所在的这个区间彻底消失，光标因此落进不再被任何激活项
  覆盖的位置，见 §2.7 的 reengage 规则(这两个动作都会调用
  `try_reengage_seek`)。

### 3.5 `show-state` 的 OSD 展示格式

首行是"位置/时长"，`pending` 的两个端点(§1.2)紧跟在同一行末尾，独立标成
`A:.. B:..`(没有值的一边显示 `--`)；完整区间列表(§1.1)从第二行开始，
每个区间单独一行，禁用的加 `(off) ` 前缀(§5)。

```
<prefix> | 00:12/01:23 | A:00:02.000 B:--
[00:08.000,00:15.000]
(off) [00:20.000,00:25.000]
```

`pending` 不会被拼成`"[a,unset]"`/`"[unset,b]"`这样的区间行、混在完整区
间列表里——早期实现是这么做的，会让人误以为未闭合区间也被塞进了同一个
列表、只是没排序，读起来像 bug；改成首行单独展示 A/B 之后不存在这个歧义
（`pending` 本来就不是一个区间，没有对应的排序位置）。

完整区间列表最多展示 `kMaxVisibleSegments`(12)行，超出时首尾各留一半、
中间用一行`"... (N more)"`代替(N 是被折叠掉的区间数)，不是把全部区间
无脑列出来——几十个区间挨个列出来会直接把 OSD 撑到超出屏幕范围。首尾各
留一半是为了同时看到最靠前和最靠后的区间(通常是当前正在关心的两端)，
不只留头或只留尾。

## 4. 尾帧冻结(原 tail-frame-extension 能力的并入)

### 4.1 结论：直接复用 tpad/apad，不需要暂停状态

`ab-loop-b` 的 pts 截断判断发生在滤镜链**之后**。也就是说，`tpad`/`apad`
补出来的克隆帧同样有真实递增的 pts，只是 `duration` 属性和 demuxer 可
seek 范围不包含这段——但原生 ab-loop 的判断根本不依赖 `duration`，只比较
实时帧 pts，所以可以把 `ab-loop-b` 直接设成"真实末尾 pts + 冻结时长"，让
原生机制**播入**(不是 seek 进)延长段后自然触发跳转。

这样设计不需要引入 `pause` 状态或自定义计时器，§2.3 的"单段自环"机制原样
适用，尾帧冻结只是一个"终点落在延长段里"的普通 active 区间，没有特殊
分支：

1. **不能显示为暂停状态**：全程正常播放，`pause` 属性不受影响。
2. **暂停时画面必须是尾帧**：延长段里每一帧都是最后一帧的克隆。
3. **继续播放后冻结从剩余时长接着走**：普通播放的暂停/继续语义，不需要
   自己记录剩余时长。
4. **总时长观感 = 真实时长 + 冻结时长**：这段区间实际播放过的时间线长度。

**唯一的实质限制**：没法直接 seek 到延长段内部的某个具体时间点(demuxer
层面这段本来就不存在)。**次要瑕疵**：`duration` 属性、以及 uosc 进度条
等依赖 `duration` 的 UI 不会把冻结时长算进去，视觉上进度条会先到 100%、
画面还继续播了一小段才真正循环——这是显示层面的落差，不影响功能是否
生效。

### 4.2 触发条件与滤镜应用

- 文件加载时按 `kTailFreezeSeconds`(默认 0.3 秒)对视频轨施加
  `lavfi=[tpad=stop_mode=clone:stop_duration=<duration>]`，有音频轨则同时
  对音频轨施加 `lavfi=[apad=pad_dur=<duration>]`。
- 任意一个 segment(或 `pending` 的开放区间)如果其"终点"语义是"循环到
  文件末尾"(即 `b` 未显式设置、或显式设置为真实 `duration` 附近)，其在
  `compute_jump_pair` 里实际写入 `ab-loop-b` 的值改为"真实末尾 pts + 冻结
  时长"，而不是真实末尾 pts 本身。
- `tail-frame-extension` 作为独立插件的存在价值收窄为"完全不使用
  enhanced-ab-loop 时的普通 `loop-file` 场景"；两者配合使用时的尾帧冻结
  能力由 enhanced-ab-loop 内部实现，不再依赖外部插件叠加生效。

## 5. 选段循环(启用/禁用子集)

- 每个完整区间(§1.1)带 `enabled: bool`，新建默认 `true`。
- `toggle-segment`：光标落在某个完整区间内部时触发，翻转该区间的
  `enabled`，触发 §2.3 重新计算 active 队列；禁用可能让光标落进不再被
  任何激活项覆盖的位置，同 §3.4 一样需要调用 `try_reengage_seek`(§2.7)。
- 只支持按位置顺序循环、跳过被禁用的区间，不支持自定义播放顺序、不做
  solo 快捷键。
- `show-state` 的 OSD 展示区分启用/禁用状态(禁用的加 `(off) ` 前缀)。
- `enabled` 字段随 §6 的存档格式一并持久化。

## 6. 存档 / 读档

- 显式触发(`Ctrl+S`/`Ctrl+L`)，不自动保存/自动加载。
- 存储内容：当前 `segments` 列表(含 `a`/`b`/`enabled`)，不存 `pending`
  (待定状态不做持久化)。
- 序列化用 nlohmann/json(单头文件、MIT 协议，`FetchContent` 在顶层
  `CMakeLists.txt` 里固定版本拉取，跟 Catch2/fmt 同一个模式；不进
  `external/`，因为不需要通读参考上游源码)。
- 两级 key：**内容采样哈希**(文件大小 + 头部采样字节 + 尾部采样字节，
  `store::compute_sample_ranges`/`compute_content_hash`，不读全文件，
  毫秒级完成)定位存档**文件**，**绝对路径的哈希**(`store::
  compute_path_hash`，不是明文路径本身)是文件内部用来区分具体条目的
  **次级 key**，不参与文件名哈希。

  ```
  ~~home/loop-segments/<内容采样哈希>.json
  {
      "entries": {
          "<路径哈希 1>": { "segments": [ {a, b, enabled}, ... ] },
          "<路径哈希 2>": { "segments": [ ... ] }
      }
  }
  ```

  **不直接存明文绝对路径**：绝对路径只留在内存里用来实际读文件、算内容
  采样哈希，写盘前会先经过一次不可逆的路径哈希(和内容采样哈希用同一套
  FNV-1a)，存档里出现的只有这个哈希。

  内容相同、路径不同的重复文件会落进**同一个**内容哈希文件，以各自路径
  的哈希分别开条目，互不覆盖、互不共享。

  - **保存**(`store::upsert`)：算出内容采样哈希 → 打开(不存在则新建)对应
    的 json → 用当前路径的哈希 upsert 一条 entry → 写回。
  - **加载**(`store::lookup`)：算出内容采样哈希 → 找到对应 json(不存在则
    视为没有存档，`kNoArchive`)→ 用当前路径的哈希精确匹配 `entries`：
    - 命中 → `kExactMatch`，直接加载。
    - 未命中，且这个内容哈希文件里只有 1 条 entry → `kSingleCandidate`，
      大概率是同一个文件被改名/挪动了。提示"发现一份内容相同、名字不同
      的存档"(不展示旧路径，插件这边也只有它的哈希)，用户按 `y`/`n`
      (`confirm-yes`/`confirm-no`，临时抢占的键，不依赖 uosc)确认后加载，
      并记录 `rename_from`，下次保存时把旧 entry 迁移成新路径的 key，
      不会留孤儿。
    - 未命中，且有 2 条及以上 entry → `kNoArchive`，属于真正的同内容不同
      名重复文件，不做自动兜底，需要用户为当前这个路径重新手动保存。
  - `kSingleCandidate` 弹出确认提示期间(等待用户按 `y`/`n`)：
    - **强制暂停**：进入确认前记一下当前 `pause` 状态
      (`paused_before_confirm`)，然后把 `pause` 设成 `true`；用户按下
      `confirm-yes`/`confirm-no` 之后，无论选哪个都把 `pause` 还原成这
      个记下来的值(不是无条件恢复播放——本来就是暂停的话不该被顺手取消
      暂停)。
    - **独占按键**：临时 `define-section`+`enable-section ... exclusive`
      一个只放行 `confirm-yes`/`confirm-no` 对应物理键的 input section，
      其余按键(含鼠标/滚轮)一律映射到 `ignore`，逼用户先回答完这条提示
      才能做别的操作。物理键不硬编码——用 `input-bindings` 属性反查当前
      实际绑定到这两个 binding 名字的键，跟用户在自己 `input.conf` 里的
      自定义保持一致。如果反查不到任何绑定(用户把这两个 binding 从
      `input.conf` 里删掉了)，**不启用**独占区段，退化成"只暂停+提示，
      不锁键"——宁可退化，也不能把用户锁死在一个连 `y`/`n` 都按不出来的
      暂停画面里出不来。
    - OSD 提示用一个 24 小时量级的超长 `duration` 模拟"常驻直到用户回
      答"(`show-text` 没有真正意义上的"永久显示"，只有一个到期时长)；
      独占按键期间不会有别的操作触发新 OSD 把它顶掉，不需要额外的定时器
      去周期性重发。
    - 文件切换(`on_file_loaded`)时如果确认还没被回答，防御性地解除独占
      区段(理论上独占键期间正常路径切换不了文件，但不排除外部 IPC 在这
      期间发 `loadfile`)。

## 8. 本轮实现踩过的坑

这一节按发现顺序记录这一轮实现/联调过程中遇到的每一个问题——包括真实的
代码 bug、一次尝试后又主动放弃的修复、以及一条被讨论澄清而不是被"修复"
的行为。每一条都尽量给出：现象、复现步骤、根因、修复方式、教训(如果要
从零重新实现，应该怎么在设计阶段就避开)。前六条(除第 1、4 条外)都已经
折算进 §2 的正文里；这里保留完整的"为什么"，供审计和以后类似设计参考。

### 8.1 TESTING.md 测试用例本身选错了坐标(不是代码 bug)

- **现象**：手动测试 A2("拒绝重叠")时，在已有 `[2,5]` 的基础上 seek 到
  3s 按 `[`，观察到的语义是"移动 A"，而不是"插入新区间"。
- **根因**：3s 严格落在已有 `[2,5]` 内部(距边界都超过 `kCursorEpsilon`)，
  按 §3.2，光标在已有完整区间内部按 `set-a`/`set-b` 触发 Move A/B 是**完全
  正确**的行为——TESTING.md 的 A2 测试选错了坐标，把 B1(Move)的触发条件
  当成了 A2(插入)的场景来测，不是代码的问题。
- **修复**：把 A2 的坐标改成 1s(在 `[2,5]` 前面的空隙)/8s。
- **教训**：写"插入新区间"类的测试用例时，涉及的每一个坐标都必须确认不
  落在任何已有区间内部(哪怕只是"顺便路过"的坐标)，否则会先触发 Move
  语义，根本走不到要测的插入流程。

### 8.2 `locate_active_index` 的兜底方向错误，导致重开循环后 ab-loop 直接失效

- **现象**：关闭循环、定义两个区间(`[2,5]`、`[10,15]`)、重新打开循环，
  循环完全不生效，一路播到文件末尾。
- **复现**：定义完 `[10,15]` 后光标恰好停在 15(刚按完 `set-b`)，此时打开
  循环。
- **根因**：`locate_active_index` 原来的兜底分支是 `return 0`(wrap 回第
  一项)。当 pos 落在"最后一项自己的 `b` 及之后"(包括恰好等于 `b`)时会
  命中这条兜底。返回 0 会让(当时还是 §8.3 那个"预置下一跳"设计的)
  `compute_jump_pair` 算出 `ab_loop_a > ab_loop_b` 的颠倒 pair；mpv 自己
  的 `get_ab_loop_times()`(§2.2)会把这个颠倒 pair 交换成"合法"区间，但
  `update_ab_loop_clip()`(`playloop.c`)比较的是**交换前**就已经确定的
  `ab_loop_clip` 标志——一旦当前 pos 已经超过交换后的有效终点，`ab_loop_
  clip` 直接判 false，原生循环整个失效。
- **修复**：兜底分支改成 `return order.size() - 1`(最后一项自己)。
- **教训**：任何"定位当前落在 active 队列哪一项"的函数，必须显式测试
  "pos 恰好等于/超过最后一项自己的终点"这个边界，不能想当然地 wrap 回
  第一项——这个边界在"刚编辑完最后一段、光标原地未动就做别的操作"时非
  常容易触发，不是罕见情况。

### 8.3 `compute_jump_pair` 的"预置下一跳"设计违反了 §2.2 的硬约束(本轮返工成本最高的一个坑)

- **现象**：多段循环里，第一段播完不会跳到第二段，会继续正常播放很久
  (甚至播过第二段的起点)，才被"倒着"拽回第一段的终点。
- **复现**：定义 `[2,5]`、`[10,15]`，取消暂停观察——期望是
  `2→5(跳)→10→15(跳)→2→...`，实际观察到的是 `2→5→(继续正常播放，经过
  10)→(继续播放到接近 15 处才触发一次倒跳)→回到 5 附近→(再往前播到接
  近 10)→倒跳回 5→...`，实测复现时用位置轮询清楚看到过这个"播过头、
  再倒跳"的完整轨迹。
- **根因**：本插件最初的 SPEC 设计是"预置下一跳"——落到某个 active 区间
  起点后，立刻把 `ab-loop-a` 设成**下一个** active 区间的起点、`ab-loop-
  b` 设成**当前**区间的终点，假设 mpv 会分别按字面属性名解释这两个值。
  但 §2.2 已经证明这个假设是错的：mpv 恒定按大小排序，跟字面属性名无关。
  正向多段轮播里"下一段起点"几乎总比"当前段终点"大，排序后完全反了——
  截止点变成了下一段起点(播过头很久才触发)，跳转目标变成了当前段自己
  的终点(倒着跳)。
- **修复**：`compute_jump_pair` 改成只写"当前落在的这一项自己"的 `a`/`b`
  (恒 `a < b`，不会被排序坑)。跨段跳转改成 §2.3 描述的显式状态机：
  `armed_index` 记录"现在原生自环围绕哪一段在转"，`on_native_landing`
  监听 `SEEK`/`PLAYBACK_RESTART`，检测到"落地位置精确等于 `armed_index`
  自己的 `a`"就判定为"这段播完一轮"，主动补发一次 `seek` 到下一段起点。
- **教训**：**这是整个实现里最值得提前预防的一条**——如果重新设计，务必
  先把 §2.2 的约束(mpv 的 ab-loop-a/b 是无序的 `[min,max]` 区间对)当成
  硬性前提写进 SPEC 草案，而不是等实现完、联调时才用 headless mpv 复现
  出来。任何"预置未来会用到的某个值"的设计，只要那个值可能比当前段的
  终点更大，就必然会撞上这条约束。

### 8.4 跨段切换瞬间的单帧闪烁——尝试修复后主动放弃，记为已知限制

- **现象**：用户观察到段与段切换的瞬间会有一次非常快的画面回跳(比如
  `[2,5]→[10,15]` 切换时，肉眼能感觉到先闪一下又跳走)。
- **复现与验证方法**：造一个"每一帧亮度按时间戳线性编码"的测试视频
  (`ffmpeg ... geq=r='(T/20)*255'`)，用 `--vo=image` 把切换瞬间**每一帧
  实际渲染输出**都存成图片，再把图片亮度还原回时间戳，重建真实显示过
  的时间序列——这样不需要肉眼判断就能确认"是否真的渲染送显过额外的
  一帧"。确认过：`compute_jump_pair`/`on_native_landing` 的自环+补跳
  机制，会在跳到下一段之前，真的渲染送显一帧"当前段自己起点"的画面。
- **根因**：`on_native_landing` 的补跳依赖"先看到原生自环已经落地在自己
  的 `a`，再反应发 `seek`"——这个反应必然滞后于 mpv 内核自己"决定跳到
  `a` 并把这一帧解码送显"这一步，因为插件是外部客户端，只能通过事件队
  列被动收到通知，天然比内核内联的逐帧判断慢半拍，无法抢在它前面。
- **尝试过的修复**：观察 `time-pos` 属性变化(mpv 对这个属性基本每帧都
  会通知一次，不是按固定间隔轮询)，在真正命中截止点前提前 1~2 帧主动
  抢跑发 `seek`。用同样的抓帧方法验证过，这个方法确实能让"额外那一帧"
  完全不出现。
- **最终决定**：**这个修复已经撤销**。用户在真实播放环境里实测，主观
  体验没有改善(推测是 mpv 精确 seek 本身固有的解码抖动，这个方法只是
  换了一种方式触发同类抖动，没有根治)，而且引入的额外状态(`cached_
  fps`/`preempt_fired`/`on_time_pos_changed`/`kPreemptFrames`/
  `within_preempt_window`)增加了不小的维护成本，权衡下来不值得，已经
  完整回退。
- **教训**：这大约 1 帧(`1/fps` 秒量级)的瞬间画面回跳，目前认定为
  **已知限制、不是应该继续投入修的 bug**(见 §9)。如果未来想再挑战它，
  已知"单纯提前触发时间点"这条路线经过验证**不能可靠解决问题**，可能
  需要更底层的方案(比如完全不使用 `ab-loop-a`/`b`，改用 mpv 的 `edl://`
  虚拟拼接多段为一个连续流)，但那意味着每次编辑区间都要重新加载文件
  (自带更明显的重载闪烁，还会丢失"循环开着的时候实时编辑"这个工作流)，
  代价远高于收益，不建议轻易尝试。

### 8.5 暂停状态下的补跳会导致 seek 死循环(本轮唯一一次真正的"死循环级"故障)

- **现象**：关闭循环、暂停播放、设置区间边界(光标停在刚设置的边界上)、
  再打开循环，mpv 状态栏显示的时间位置疯狂跳动，长时间不停。
- **复现**：headless mpv + IPC 复现实测，2 秒内触发了两万多次 `seek`。
- **根因**：见 §2.4——mpv 对"插件自己发出的显式 seek"恒定发 `SEEK` +
  `PLAYBACK_RESTART` 两条事件，原生自环只发一条 `SEEK`。原来的代码收到
  第一条事件就确认落地、清掉 `pending_self_seek`，导致第二条事件(其实是
  同一次 seek 的回声)到达时，标记已经是 false，而光标位置跟"这一段自己
  的起点"分毫不差——这跟"原生自环真的又完整播完一轮"长得一模一样，于是
  被误判成一次新的完成，触发又一次补跳，如此往复。**播放中不会露馅**，
  是因为两条事件之间总有真实播放时间流逝，光标已经往前挪了一点，恰好
  躲过"精确等于起点"这个判定；**暂停状态下光标完全冻结**，这个误判条件
  恒成立。
- **修复**：`pending_self_seek` 必须同时等到 `SEEK` 和 `PLAYBACK_RESTART`
  都收到才清掉，中途收到的 `SEEK` 先按兵不动。
- **教训**：**任何基于"事件通知"做状态判断的逻辑，都要先搞清楚同一个
  操作在 mpv 里到底会触发几条事件，不能假设"一次动作 = 一条通知"**，
  尤其是"插件自己发起的动作"和"mpv 内核自己触发的动作"很可能事件数量
  不对称。**用暂停状态测试**是发现这类 bug 最有效的手段之一——它消除了
  "真实时间流逝"这个天然会掩盖竞态问题的因素，让原本"侥幸能跑"的代码
  路径稳定复现出问题。

### 8.6 五个"可能让光标孤立"的入口，是分四轮才补全的(重开循环 / 手动 seek / 撤销端点 / 禁用区间 / 加载存档)

这五个问题本质上是同一条规则(§2.7)被遗漏在不同入口，分四次报告、四次
修复才补全：

- **8.6a 重新打开循环**：关闭循环期间光标停到没有被任何区间覆盖的位置，
  重新打开循环不会主动 seek 进循环范围。根因同 §8.2 提到的
  `update_ab_loop_clip()` 转义机制——一旦当前 pos 已经超过目标终点就直接
  禁用 clip。**修复**：新增 `loop_reengage_target`，在 `on_toggle_
  enabled` 里调用。
- **8.6b 手动 seek 落空隙**：`loop_reengage_target` 一开始只在
  `on_toggle_enabled` 里调用。已有 `[2,5]`、`[10,15]`，循环已经开着，手动
  把播放进度拖到 7(两段之间的空隙)，松手后继续正常播放，而不是立刻
  跳进 10——一路播过空隙、播过下一段的起点，直到碰到下一段的终点才被
  原生机制拽回来。**修复**：抽出 `try_reengage_seek` 辅助函数，也在
  `on_native_landing` 判定"不是原生自环播完"之后调用。
- **8.6c 撤销端点/禁用区间**：`try_reengage_seek` 泛化之后，`on_unset_a`/
  `on_unset_b`/`on_toggle_segment` 仍然只调用 `refresh_loop`。定义
  `[2,3]`、`[11,12]`，光标定位到 `[2,3]` 内部，撤销 A 再撤销 B(`[2,3]`
  彻底消失，只剩 `[11,12]`)，继续播放不会跳进 `[11,12]`。**修复**：这
  三个函数也都改用 `try_reengage_seek`。
- **8.6d 加载存档(load-loops)**：`apply_loaded_segments`(被 `on_load_
  loops` 的 `kExactMatch` 分支和 `on_confirm` 的确认分支共用)从头到尾
  只调用 `refresh_loop`，没有走 `try_reengage_seek`。现象：先定义
  `[10,15]` 并 `Ctrl+S` 保存，重新打开同一个文件(内存里的 `segments`
  被 `on_file_loaded` 清空)，光标随便停在哪(比如 2s，不在 `[10,15]`
  里)，按 `Ctrl+L` 加载出 `[10,15]`——`show-state` 显示区间确实加载对
  了，但播放照常从 2s 往前继续，不会立刻跳进 `[10,15]` 开始循环，直到
  播放自己推进到 10s 才会被原生机制接管。跟 8.6a/8.6b/8.6c 是完全一样
  的根因："改变了 `active_order` 之后没有检查光标还在不在里面"，只是这
  次的入口是"加载出全新的 `segments`"而不是"删掉/禁用已有的"。**修复**：
  `apply_loaded_segments` 里 `refresh_loop` 之前先 `build_active_order`
  + `try_reengage_seek`，吸入不了(比如 `loop_enabled` 是 `false`)才退回
  `refresh_loop`。
- **教训**：**"光标当前有没有被任何激活区间覆盖"是一个贯穿多个操作的
  通用状态，不应该只在某一个具体动作里特判**。任何会改变
  `active_order`(增删/启用禁用 segment，`pending` 出现/消失，**或者整个
  `segments` 被替换**)或者移动光标的操作，完成后都应该统一问一遍："这
  样做完之后光标还在激活范围内吗？不在的话要不要主动吸入？"——如果一
  开始设计时就把这条规则写成 §2.7 这样一条独立的通用规则(而不是留给每
  个动作各自决定"我要不要也管这个")，能一次把五个入口都做对，不需要跟
  着 bug 报告一个个补；这次 8.6d 就是把"增删/启用禁用"想全了、却漏了
  "整个替换"这一种同样会改变 `active_order` 的方式。

### 8.7 "落在空隙就立刻吸入"与"任何时候都能设置端点"之间的张力(讨论澄清，不是 bug)

- **现象**：把 §8.6b 的修复从"只在打开循环时生效"泛化到"任何手动 seek
  落在空隙都吸入"之后，发现"seek 到一个全新的空白位置去定义第二个区间"
  这个操作，如果循环还开着，也会被立刻拽走——比如已经有 `[2,5]`，循环
  开着时 seek 到 10 准备按 `[` 定义第二段，还没来得及按键，光标已经先
  被拽回 2 了。
- **讨论结果**：§1.3 的表述("任何时候都可以设置/调整端点，不受
  `loop_enabled` 影响")本身没有错——`set-a`/`set-b` 这两个动作确实随时
  能按、不受影响。但"先把光标 seek 到目标位置"这个前置步骤，不在这条
  保证的覆盖范围内，会被 §2.7 的"落在空隙就吸入"规则打断。也就是说，
  循环开着的时候，实际上只能顺利维护"光标已经站在里面"的那一段(Move
  A/B、nudge)，没法顺手在空白处插入一个全新的区间。
- **结论**：**保留 §2.7 现有行为**(不缩小它的作用范围)，改为在
  TESTING.md 里明确写清楚使用约定："定义第二段及以后的区间前，先按 `|`
  关掉循环"。这是本插件对"编辑多段区间"这个工作流的**有意限制**，不是
  需要修的 bug。
- **教训**：给一条"任何时候都可以 X"的规则写 SPEC 时，要想清楚"做 X 之
  前需要先导航到某个位置"这个前置动作，是不是也应该在"任何时候"的保护
  范围内——如果范围不覆盖它，一定要在 SPEC 里明确写出这个限制，不要留
  给以后靠 bug 报告才发现。

## 9. 已知限制(不是 bug，测到这些不用报)

- **跨段切换瞬间约 1 帧(`1/fps` 秒量级)的画面回跳**：见 §8.4，已尝试修复
  又主动放弃，认定为架构性代价，不是待修的 bug。
- **循环开着的时候没法顺手 seek 到空白处插入新区间**：见 §8.7，是 §2.7
  吸入规则的自然结果，不是 bug；工作流上请先关循环再定义新区间。
- 没法把进度条/章节跳转/`seek` 命令直接拖到尾帧冻结延长出来的那一小段
  时间里(demuxer 层面这段本来就不存在)。
- 进度条/剩余时间这类基于 `duration` 属性的显示不会把冻结时长算进去，
  播完真实内容后进度条会先显示 100%，画面还会继续"播"一小会儿。
- uosc 进度条目前只会展示"当前激活的一段"(原生 `ab-loop-a`/
  `ab-loop-b`)，还看不到全部多段区间——见 §7。
- §6 存档消歧遇到"两个以上内容相同的候选"时不会弹出候选列表让你选，
  直接按"没有存档"处理，需要手动为当前文件重新存一份。
