# enhanced-ab-loop 重写 SPEC

本文档是 `enhanced-ab-loop`(以及并入的 `tail-frame-extension`)重写前的设计
草案,记录讨论过程中定下来的行为约定,供实现前审计、实现中对照。不是最终
用户文档——重写完成后应该按 [AGENTS.md](../../AGENTS.md) 的要求另外整理
`README.md`。

本文档只覆盖行为设计,不包含具体代码结构,语言选型见「未决问题」一节。

## 0. 背景与目标

现状(`plugins/enhanced-ab-loop/lua/enhanced-ab-loop.lua`)是纯 Lua 实现:

- 用 `mp.add_periodic_timer(0.05, check)` 轮询 `time-pos` 判断是否越过循环
  终点,精度受 50ms 定时器粒度和脚本事件循环调度双重制约。
- 完全没有使用 mpv 原生的 `ab-loop-a`/`ab-loop-b`/`ab-loop-count` 属性。
- 循环终点无法设到 `tail-frame-extension` 用 `tpad`/`apad` 滤镜补出来的
  尾部延长段里,因为 `duration` 属性和 demuxer 可 seek 范围不受滤镜链影响。
- `sort_segments` 的比较函数用 epsilon 容差做 tie-break,不满足严格弱序,
  `table.sort` 在这种情况下不保证正确排序。
- `eof-reached` 处理器硬编码跳到 `active[1].a`(整体列表里最靠前的一段)。

重写目标:

1. 用 mpv 原生 `ab-loop-a`/`ab-loop-b` 的帧级 pts 比较驱动跳转,替代 Lua
   定时器轮询,同时支持多段不相交区间的顺序循环。
2. 把"循环到文件真实末尾时冻结尾帧"的能力并入本插件,替代
   `tail-frame-extension` 原来单独存在但无法与 AB-loop 配合的状态。
3. 修掉排序、eof 跳转目标这两个已确认的 bug。
4. 支持挑选已有区间的子集参与循环(启用/禁用,按位置顺序)。
5. 支持把区间列表存档/读档。
6. 明确"插入新区间"和"调整已有区间边界"这两类操作的边界,避免语义模糊。

## 1. 数据模型

### 1.1 完整区间(segment)

```
segment = {
    a: double,       -- 起点,单位秒,直接存 mpv 报告的原始 double,不做量化
    b: double,       -- 终点,同上
    enabled: bool,    -- 是否参与循环(默认 true),见 §5
}
```

`segments` 是这些完整区间的列表,任何时刻都保持互不重叠、按 `a` 升序排列。

**精度处理**(对应讨论中的结论):

- 不引入帧号或定点整数量化。double 本身精度远超任何实际帧时长量级
  (`playloop.c:1130-1135` 证实有视频轨道播放时 `time-pos` 直接取当前解码/
  滤镜输出帧自身的 pts,不在两帧之间插值,取到的值本身就是某一帧的真实
  时间戳,不存在"取值不确定"的问题)。
- **存储值之间的比较**(排序、检测两个已定义 segment 是否重叠)一律用精确
  相等/大小比较,不用 epsilon。这同时修掉了现有 `sort_segments` 因 epsilon
  tie-break 不满足传递性导致排序错误的 bug——新的排序只需要 `lhs.a <
  rhs.a`,不需要 tie-break(正常状态下不会有两个不同 segment 的 `a` 精确
  相等)。
- **实时播放位置 vs 已存储边界的比较**(判断光标当前是否"落在"某个已有
  完整区间内,用来触发 §3.2 的 Move A/B 或 §3.1 的"插入被拒绝"判定)保留一
  个固定的、故意放宽的 epsilon 常量(量级参考现有配置 `epsilon = 0.005`),
  这是给键盘操作留出的人体工学余量,不是精度补偿。

### 1.2 待定端点(pending point)

同一时刻最多存在一个:

```
pending = { a: double, b: nil } | { a: nil, b: double } | nil
```

对应"只设了一个端点、还没配对成完整区间"的状态。这不是一个独立的模式
标志位,只是 `A`/`B` 两个变量中恰好一个非空——讨论里提到的「编辑模式」
就是指这个状态,不需要额外的 `editing` 布尔量。

### 1.3 循环开关

`loop_enabled: bool`,由 `|` 键(`toggle-enabled`)控制,与 `pending`/
`segments` 完全独立:

- 任何时候都可以设置/调整端点,不受 `loop_enabled` 影响。
- `loop_enabled = false` 时不发生任何自动跳转(包括对 `pending` 中"开放
  区间"的实时循环),但已经设置的端点、`segments` 内容保持不变。
- `loop_enabled = true` 时,`segments` 中 `enabled = true` 的区间参与循环,
  `pending`(如果存在)也会作为一个临时的"开放区间"实时参与,见 §2.3。

## 2. 核心跳转机制

### 2.1 原生驱动,取代定时器轮询

不再用 `add_periodic_timer` 轮询判断,改为动态写 mpv 原生
`ab-loop-a`/`ab-loop-b` 属性,让 mpv 内核用帧级 pts 比较触发跳转:

- `playloop.c:927-933`(`handle_loop_file`)在到达 EOF 判定时,读取**此刻**
  的 `ab-loop-a` 实时值作为 seek 目标,`MPSEEK_EXACT` 精确 seek。
- `video.c:517-521` / `audio.c:691-695` 对每一帧比较 `pts >= ab-loop-b`
  实时值,一旦命中该帧直接判 EOF(不渲染)。
- 两处都是"触发那一刻才读当前属性值",不是缓存值。

**预置下一跳**:利用这个"实时读取"的特性,不等跳转发生后再补救,而是在
每次落到某个 active 区间起点后,立刻把 `ab-loop-a` 预置为**下一个** active
区间的起点、`ab-loop-b` 预置为**当前** active 区间的终点。这样命中终点时
mpv 直接精确 seek 到下一段起点,不会闪现一帧再补跳。

落地流程:

1. `active_loop_order`:`segments` 中 `enabled = true` 的区间,按位置排序,
   如果存在 `pending` 则追加为最后一项(见 §2.3 的临时边界规则)。
2. 通过 `seek`/`playback-restart` mpv 事件确认"已经落到 active_loop_order
   中第 i 项的起点"后,立即设置:
   - `ab-loop-a` = `active_loop_order[(i+1) % n].a`
   - `ab-loop-b` = `active_loop_order[i].b`
   - `ab-loop-count` 保持默认 `inf`,不需要按段管理计数
     (`misc.c:132`:`remaining_ab_loops` 对 inf 值不递减)。
3. `active_loop_order` 发生变化(增删 segment、切换 enabled、
   `pending` 出现/消失)时,需要重新计算当前所在项并重新预置下一跳,不能
   只在落地事件里更新。
4. `loop_enabled = false` 时清空 `ab-loop-a`/`ab-loop-b`(unset,不是设成
   相等值),让原生机制完全不介入;重新置 `true` 时按当前光标位置重新
   计算并设置。

### 2.2 eof-reached 兜底

`get_ab_loop_times`(`misc.c:124-141`)要求 A、B 都必须是有效数值,`pending`
只有一个端点时无法直接表达给原生机制,所以 `pending` 只有一个端点、且不存
在下一个 active 区间兜底时(即 `pending` 是 `active_loop_order` 中最后一项
且 `b` 未定),会真正播到文件末尾触发 `eof-reached`,而不是被原生 pts 截断
拦下。

现有实现里这个回调硬编码跳 `active[1].a`(整体第一段)。修法:改成
`active_loop_order[#active_loop_order].a`(队列中最后一项,即真正播到文件
末尾的那一项自己的起点)。这个改动经过验证是严格优于现状的:

- 只有单一未闭合端点、没有任何完整区间时,`active_loop_order` 只有一项,
  两种写法结果相同,不影响现有"不会跳回字面 0"的约束。
- 有完整区间存在时,新写法跳回的是"刚播完的那一段自己的起点",而不是
  无论如何都绕回最前面一段——语义更直觉,也不会在编辑靠后的区间时被拖
  回最前面重新播一轮。

### 2.3 pending 的临时循环边界

`pending` 存在时(§1.2),需要一个临时边界让它在编辑过程中也能实时循环
(不能等到真正设置第二个端点才开始循环,用户需要靠反复播放来找到另一个
端点该落在哪)：

- 只有 `a`、没有 `b`:临时终点 = **下一个**完整区间的**起点**(不是终点),
  如果不存在下一个完整区间,临时终点为空(即循环到文件真实末尾,由
  §2.2 的 eof-reached 兜底处理)。
- 只有 `b`、没有 `a`:临时起点 = **上一个**完整区间的**终点**(不是起点),
  如果不存在上一个完整区间,临时起点为 `0`。

这里特意用"近端"(下一个区间的起点/上一个区间的终点)而不是"远端",是
为了避免编辑中的开放区间一路播穿整个相邻的已有区间——只应该在相邻区间
开始之前打转,不应该吞掉相邻区间本身的循环范围。

这个临时边界**只影响 pending 状态下的实时循环范围**,不影响它最终配对
成完整区间时的实际取值(那部分见 §3.1)。

## 3. 插入 / 调整区间的交互

### 3.1 插入新区间(set-a / set-b)

`set-a`/`set-b` 在光标不落在任何已有完整区间内部时触发,语义**始终且仅
表示"插入一个独立的新区间"**,不做任何隐式的边界借用/吞并:

- 第一次按(比如 `set-a`)记录 `pending.a`。
- 第二次按另一个键(`set-b`)尝试配对成完整区间,校验:
  - `a < b`(否则拒绝,提示非法)。
  - 与 `segments` 中任意已有完整区间不重叠(`overlaps_complete`,精确比较,
    不用 epsilon)。命中重叠直接拒绝、提示冲突,不做任何自动收缩/clamp/
    忽略邻居的行为。
- 校验通过才真正插入 `segments`,按 `a` 排序,清空 `pending`。

如果最终两个端点落在已有区间内部导致重叠,用户需要自己把端点往回调再
重试,插件不代为决策。

### 3.2 调整已有区间边界

两类相邻但不同的操作,**必须用不同的触发方式区分**,不能用同一个手势
靠事后结果反推意图(这是重写前反复讨论确认过的地方,现有实现在这里语义
模糊,修的时候要连带把交互也理清楚):

- **Move A / Move B**(已有功能,对齐 PotPlayer,继续保留):光标落在某个
  已有完整区间**内部**时按 `set-a`/`set-b`,直接把该区间自己的 `a`/`b`
  移动到光标位置(前提是移动后不会和相邻区间冲突、不会导致区间反转)。
- **Extend segment**(新增,`extend-prev`/`extend-next` 两个独立键位,不
  复用 set-a/set-b):光标落在空隙(不在任何完整区间内部)时触发,拉伸
  相邻区间的近端边界到光标位置。用两个方向各自独立的键位,不靠"猜哪个
  邻居更近"这种隐式判定,理由和具体规则见下。

  - `extend-prev`:把光标所在空隙**前面**那个已有区间的 `b` 拉伸到光标
    位置;空隙前面没有区间(光标在第一段之前)则拒绝,提示"没有前一段
    可拉伸"。
  - `extend-next`:把光标所在空隙**后面**那个已有区间的 `a` 拉伸到光标
    位置;空隙后面没有区间则拒绝,提示"没有下一段可拉伸"。
  - 目标位置如果会和**再远一层**的区间冲突(比如拉伸前一段的 `b` 超过了
    后一段的 `a`),直接拒绝、提示冲突,不做自动收缩/clamp——和 §3.1
    "冲突就拒绝,不做隐式处理"保持同一条总原则。
  - `pending`(§1.2)非空时(正在插入一段还没配对完成),extend 直接拒绝,
    提示需要先完成或放弃当前插入——同一时间只允许一种进行中的编辑,不
    允许插入和 extend 状态叠加,这个约束和现有 Move A/B 的前提条件
    (`pending` 为空)是同一种写法,不是新规则。
  - 拉伸只会让边界往空隙方向移动,不会导致 `segments` 顺序颠倒,成功后
    不需要重排,但仍需触发 §2.1 第 3 点"重新计算 active 队列 + 预置下
    一跳"。

  这样设计之后,extend 和其余三类操作(插入、Move、pending 中的插入)靠
  **触发前提互斥**天然不冲突,不需要任何时候靠"事后看结果反推意图":
  set-a/set-b 永远只处理插入;Move A/B 的前提是光标在区间内部;extend 的
  前提是光标在空隙里;三者的触发条件两两不重叠,同一个光标位置只可能
  命中其中一种。

### 3.3 nudge

现有的 `nudge-a-*`/`nudge-b-*`(按固定步长微调光标所在完整区间的端点)
保留,但相对现状有两处修正:

- **触发条件放宽**:现有实现复用 Move A/B 的 `is_inside_segment` 判定,
  要求光标距区间两端都超过 `epsilon` 才算"在区间内"。这个严格判定是
  Move A/B 用来区分"光标在区间内部"和"光标在空隙里"这两种语义需要的,
  nudge 不存在这种二义性(永远是在调整一个已有区间),不需要这条限制。
  nudge 的触发条件改成:光标落在 `[a, b]` 闭区间内即可,不再要求距边界
  超过 epsilon——否则循环刚跳回 `a`/`b` 附近(原生 `MPSEEK_EXACT` 落点
  通常就在边界几毫秒内)时,恰恰是最想 nudge 的时机,却会被拒绝。
- **顶到边界时拒绝,不做静默 clamp**:现有实现顶到边界会静默把值压到
  能压到的最大值,不提示。改成和 §3.1/§3.2 一致的"顶到边界就拒绝这次
  nudge、提示已到边界",不做隐式收缩,四类调整区间的操作(插入/Move/
  extend/nudge)统一遵守同一条"冲突就拒绝"的规则。

调整成功后需要触发 §2.1 第 3 点的"重新计算并预置下一跳"。

### 3.4 撤销端点(unset-a / unset-b)

保留现有语义(把某个完整区间拆回一个 `pending`),但显示逻辑要一并修掉:
拆出来的 `pending` 必须按 §1.2/§2.3 的规则独立展示为"[unset, b]" 或
"[a, unset]",不能像现有实现那样把它错误地嫁接到相邻完整区间上、篡改
相邻区间自己的显示。

## 4. 尾帧冻结(原 tail-frame-extension 能力的并入)

### 4.1 结论:直接复用 tpad/apad,不需要暂停状态

`ab-loop-b` 的 pts 截断判断发生在滤镜链**之后**
(`video.c:504,517-518` 读的是 `vo_chain->filter` 的输出;`audio.c:687-695`
同构)。也就是说 `tpad`/`apad` 补出来的克隆帧同样有真实递增的 pts,只是
`duration` 属性和 demuxer 可 seek 范围不包含这段——但原生 ab-loop 的判断
根本不依赖 `duration`,只比较实时帧 pts,所以可以把 `ab-loop-b` 直接设成
"真实末尾 pts + 冻结时长",让原生机制**播入**(不是 seek 进)延长段后自然
触发跳转。

这样设计不需要引入 `pause` 状态或自定义计时器,§2.1 描述的"预置下一跳"
机制原样适用,尾帧冻结只是一个"终点落在延长段里"的普通 active 区间,没
有特殊分支。已验证满足全部约束:

1. **不能显示为暂停状态**:全程正常播放,`pause` 属性不受影响。
2. **暂停时画面必须是尾帧**:延长段里每一帧都是最后一帧的克隆,不管在
   延长段哪个位置暂停,画面都是同一张尾帧图。
3. **继续播放后冻结从剩余时长接着走**:这就是普通播放的暂停/继续语义,
   不需要自己记录剩余时长。
4. **总时长观感 = 真实时长 + 冻结时长**:直接成立,是这段区间实际播放
   过的时间线长度。

**唯一的实质限制**:没法直接 seek 到延长段内部的某个具体时间点(拖进度条/
章节跳转/`seek` 命令都是 demuxer 层面的操作,延长段在 demuxer 层面不存
在)。这不影响循环跳转本身(跳转目标永远是下一段的起点,在真实内容范围
内),只影响"用户主动想跳进延长段内部某处"这种本来就不常见的操作。

**次要瑕疵**:`duration` 属性、以及未来 uosc 进度条等依赖 `duration` 的
UI 不会把冻结时长算进去,视觉上进度条会先到 100%、画面还继续播了一小段
才真正循环——这是显示层面的落差,不影响功能是否生效。

### 4.2 触发条件与滤镜应用

- 沿用现有 `tail_frame_extension` 配置的思路:文件加载时按配置的
  `duration`(冻结时长)对视频轨施加
  `lavfi=[tpad=stop_mode=clone:stop_duration=<duration>]`,有音频轨则同时
  对音频轨施加 `lavfi=[apad=pad_dur=<duration>]`(与现有
  `plugins/tail-frame-extension/lua/tail-frame-extension.lua` 的做法一致,
  直接把这部分逻辑迁移过来)。
- 任意一个 segment(或 `pending` 的开放区间)如果其"终点"语义是"循环到
  文件末尾"(即 `b` 未显式设置、或显式设置为真实 `duration`),其在
  §2.1 中实际写入 `ab-loop-b` 的值改为"真实末尾 pts + 冻结时长",而不是
  真实末尾 pts 本身。
- `tail-frame-extension` 作为独立插件的存在价值收窄为"完全不使用
  enhanced-ab-loop 时的普通 `loop-file` 场景";两者配合使用时的尾帧冻结
  能力由 enhanced-ab-loop 内部实现,不再依赖外部插件叠加生效。

## 5. 选段循环(启用/禁用子集)

- 每个完整区间(§1.1)带 `enabled: bool`,新建默认 `true`。
- 新增键位 `toggle-segment`:光标落在某个完整区间内部时触发,翻转该区间
  的 `enabled`,并触发 §2.1 第 3 点重新计算 active 队列。
- 只支持按位置顺序循环、跳过被禁用的区间,不支持自定义播放顺序、不做
  solo 快捷键(先做简单版本,以后有需要再加)。
- OSD 展示需要区分启用/禁用状态(比如禁用的加前缀或改变文字样式),具体
  展示方式实现时定。
- `enabled` 字段随 §6 的存档格式一并持久化。

## 6. 存档 / 读档

- 显式触发,不自动保存/自动加载。
- 存储内容:当前 `segments` 列表(含 `a`/`b`/`enabled`),不存 `pending`
  (待定状态不做持久化)。
- 序列化用 [nlohmann/json](https://github.com/nlohmann/json)(单头文件、
  MIT 协议)。`mp.utils.format_json`/`parse_json` 是 Lua 脚本层 `mp.utils`
  模块提供的能力,C 插件只有 `mpv/client.h` 这层 client API,没有对应的
  JSON 序列化函数,不能用。按 AGENTS.md「`external/` 只放需要通读参考的
  上游源码」的约定,nlohmann/json 不进 `external/`,用 `FetchContent` 在
  顶层 `CMakeLists.txt` 里固定版本拉取,和 Catch2/fmt 同一个模式。选它
  而不是 YAML/TOML 的原因:TOML 的分节/table key 语法不适合"任意文件
  路径做 key 的嵌套 map"这种 schema;YAML 语法本身有不少隐式类型转换的
  坑(`no`/`yes` 之类被解析成布尔值),要写得可靠得引入比 nlohmann/json
  重得多的 `yaml-cpp`,换来的"人工编辑更友好"在"插件自动写、偶尔手动看
  一眼"这个场景里只是加分项,不是刚需;JSON 语法小而无歧义,生态里的
  单头文件库又足够成熟,是这个场景性价比最高的选择。
- 两级 key:**内容采样哈希**(文件大小 + 头部采样字节 + 尾部采样字节,
  不读全文件,毫秒级完成)定位存档**文件**,**绝对路径**是文件内部用来
  区分具体条目的**次级 key**,不参与文件名哈希。

  ```
  ~~/loop-segments/<内容采样哈希>.json
  {
      "entries": {
          "<绝对路径 1>": { "segments": [ {a, b, enabled}, ... ] },
          "<绝对路径 2>": { "segments": [ ... ] }
      }
  }
  ```

  内容相同、路径不同的重复文件(你提到的档案库场景)会落进**同一个**
  内容哈希文件,以各自路径分别开条目,互不覆盖、互不共享。

  - **保存**:算出内容采样哈希 → 打开(不存在则新建)对应的 json → 用当前
    绝对路径 upsert 一条 entry → 写回。
  - **加载**:算出内容采样哈希 → 找到对应 json(不存在则视为没有存档,
    结束)→ 用当前绝对路径精确匹配 `entries`:
    - 命中 → 直接加载,没有任何歧义(路径是精确 key,不是模糊候选)。
    - 未命中 → 看这个内容哈希文件里一共有几条 entry:
      - **只有 1 条**:大概率是同一个文件被改名/挪动了,不是真的重复
        文件。此时提示"找到内容相同但路径不同的存档,是否要读取并重新
        关联到当前路径",用户确认后加载,并在下次保存时把旧路径的 entry
        迁移/覆盖成新路径。mpv 本身没有原生弹窗/对话框 API(`client.h`
        没有 messagebox 类函数,uosc 的菜单也是脚本自己拿 OSD/ASS 画出来
        的,不是系统对话框),这个确认要靠"显示 OSD 提示文字 + 临时抢占
        一两个按键(比如 Y/N)当确认/取消、用完恢复原按键"来实现,不依赖
        uosc。
      - **2 条及以上**:属于真正的同内容不同名重复文件,无法自动判断
        该用哪一条,不做自动兜底,按"没有存档"处理,需要用户为当前这个
        路径重新手动保存(v1 先不做"列出全部候选让用户手动选"这种
        picker,复杂度留到有实际需要时再加)。

  这样"路径做二次筛选"不再是一个需要临场提示候选列表的模糊匹配,而是
  文件内部的精确 key 查找,只有"单一候选、疑似改名"这一种情况需要用户
  确认,§8 里原本这条未决问题可以去掉了。
